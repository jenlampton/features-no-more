How does the Features module do it?

Drupal's API!

The API provides ways to ingest things* from code.

(*these are all the things that work reliably with Features)

I can do that!

Use the API in the way it was intended: write a module.

But, what about all the other things?

Drupal expects some configuratons to be in the database and not in code.

Here's a crazy idea...

Can we leave them in the database?

Two ways to manage config in the database:

  1. Do things manually (if it's safe)
  2. Wtite an update hook (if it's not)

Way #1: Doin' it manually

  1. Take good notes, then follow them.
    • document everything that needs to be repeated, then repeat them (suitable only for simple tasks)
  2. Create new things directly on the live site.
    • With the necessary access controls in place, they won't be noticed. (not for the faint of heart)

Way #2: Updain' it oldskool-like

Write an update hook.

If you need to modify something only at the time of deployment, this is your ticket.

OMG!

CUSTOM CODE!!!

Let's reveiw the goal:

  • Don't harm site performace
  • Save developer time (and client $)
  • Retain developer sanity

I've been able to achieve all of these goals reliably
using custom code.

What's in the code?

  • A module for each type of configuration that can be cleanly exported: Contains organized, individual, files for each export.
  • And (if you want) a custom module specifically for updates at the time of deployment.
